# 配置区选择框与默认配置恢复设计

## 背景
设备装载量配置区域当前存在两个问题：
1. “删除选中”功能不可用，现有实现依赖表格隐式选择状态，不满足稳定交互需求。
2. “保存配置”只能走当前默认配置保存路径，用户无法自定义 JSON 文件名称和保存位置。

基于新的确认，本次需求进一步明确为：
- 表格每一行都要有一个明确可见的选择框。
- 保存配置时允许用户选择 JSON 文件名和保存位置。
- 支持从项目内置默认配置恢复到界面。
- 恢复默认配置只影响界面，不自动写回磁盘文件。

## 目标
1. 在设备装载量配置表格中增加显式“选择”列，每行都有可见选择框。
2. 修复“删除选中”，支持按勾选状态批量删除。
3. “保存配置”改为另存为 JSON，由用户选择保存位置和配置名称。
4. 增加“恢复默认配置”按钮，从项目内置 `config.json` 恢复到界面。
5. 恢复默认配置后不自动持久化，只有用户点击保存时才写出文件。

## 非目标
1. 不修改项目内置默认 `config.json` 的结构。
2. 不引入“用户自定义默认配置”机制。
3. 不重构整个 GUI，只调整配置区内部实现。
4. 不扩展 Excel/CSV 等其他导出格式。

## 方案选择

### 方案 A（采用）
- 在表格中增加显式选择框列。
- 用轻量数据结构保存配置区状态，而不是直接以 `DataRow` 作为唯一状态源。
- 点击“删除选中”时按这份数据状态删除并重建表格。
- “保存配置”通过保存文件对话框导出 JSON。
- 新增“恢复默认配置”按钮，从项目内置默认配置读取并刷新界面。

采用原因：最符合当前交互需求，行为清晰、可维护性最好。

### 方案 B（未采用）
- 继续依赖 `DataTable` 自带选择列，只修复状态同步。
- 额外增加另存为 JSON 和恢复默认配置。

未采用原因：不满足“每一行增加一个可见选择框”的明确要求。

### 方案 C（未采用）
- 每行提供单独删除按钮，不支持批量选择。
- “保存配置”支持另存为 JSON。

未采用原因：不满足批量“删除选中”的目标。

## 详细设计

### 1. 配置区界面结构调整
配置表新增三列：
- `选择`
- `设备型号`
- `装载量(吨)`

其中 `选择` 列使用明确可见的复选框控件。

按钮区调整为：
- `添加设备`
- `删除选中`
- `导入配置`
- `恢复默认配置`
- `保存配置`

交互要求：
- 勾选框用于控制当前行是否参与批量删除。
- “恢复默认配置”只刷新当前表格。
- “保存配置”通过文件保存对话框决定 JSON 输出路径和文件名。

### 2. 配置区状态管理
配置区内部增加一层轻量状态表示，每行至少包含：
- `selected: bool`
- `device: str`
- `capacity: str | int`

行为规则：
1. 表格由当前状态列表渲染得到。
2. 勾选框变化时，更新对应行的 `selected`。
3. 文本框变化时，更新对应行的 `device` 或 `capacity`。
4. 删除选中时，过滤掉 `selected=True` 的行并重新渲染。
5. 加载默认配置、导入配置、新增设备都通过同一套状态构建逻辑进入表格。

这样可以避免把 Flet 组件实例本身当作唯一状态来源，从而让删除、恢复、保存逻辑保持一致。

### 3. 保存配置流程
点击“保存配置”后：
1. 调用 Flet 保存文件对话框。
2. 提供默认文件名，例如 `config.json`。
3. 用户选择保存目录和文件名。
4. 将当前界面状态整理为 JSON：

```json
{
  "device_load_map": {
    "NTE240": 85,
    "TR100": 35
  }
}
```

5. 写入用户选择的目标路径。
6. 成功后记录日志，输出保存路径。
7. 如果用户取消，不报错、不修改当前界面状态。

### 4. 恢复默认配置流程
点击“恢复默认配置”后：
1. 读取项目内置默认配置文件 `config.json`。
2. 解析其中的 `device_load_map`。
3. 用默认配置内容重建当前配置区界面状态。
4. 更新表格显示。
5. 不自动写入磁盘上的任何目标文件。

恢复后的状态只存在于当前界面，用户如果需要保存，必须主动点击“保存配置”。

### 5. 错误处理
- 若默认配置文件读取失败，记录日志并保持当前界面不变。
- 若导入的 JSON 不包含 `device_load_map`，记录日志并不覆盖当前界面。
- 若保存时用户取消，静默返回，不报错。
- 若装载量字段不是合法整数，保存时继续记录日志警告并跳过无效项或阻止输出（实现时选一种并写入测试，优先延续当前“记录警告并跳过无效项”的行为）。

### 6. 测试设计（TDD）
先写失败测试，再做实现。

#### 配置区渲染测试
覆盖点：
1. 每一行都有显式选择框控件。
2. 设备型号和装载量能从状态正确渲染到界面。

#### 删除选中测试
覆盖点：
1. 勾选一行或多行后，删除只移除被勾选的行。
2. 未勾选的行保持不变。

#### 恢复默认配置测试
覆盖点：
1. 恢复后界面状态与项目内置默认 `config.json` 一致。
2. 恢复动作不直接写文件。

#### 保存配置测试
覆盖点：
1. 当前界面状态会序列化成正确的 `device_load_map` JSON 结构。
2. 保存文件路径由用户选择。
3. 用户取消保存时不报错、不写文件。

## 验收标准
1. 配置区表格每一行有明确可见的选择框。
2. “删除选中”可正确批量删除勾选项。
3. “恢复默认配置”可将界面恢复到项目内置 `config.json` 内容。
4. 恢复默认配置不会自动写回磁盘。
5. “保存配置”允许用户自定义 JSON 文件名和保存路径。
6. 相关自动化测试存在，并能覆盖上述行为。
